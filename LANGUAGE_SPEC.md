# LANGUAGE_SPEC.md

Це специфікація мови програмування РАВЛИК у її поточному стані та з позначенням місць, які треба уточнити або виправити. Документ має бути джерелом істини для parser/runtime/manual/tests.

## 1. Призначення мови

РАВЛИК — навчальна текстова мова для дітей. Вона має допомогти перейти від блокового програмування до текстового коду через зрозумілі українські команди, миттєвий візуальний результат і дружні помилки.

Принципи мови:

- простота важливіша за універсальність;
- передбачуваність важливіша за “магічність”;
- кожна помилка має бути пояснена людською мовою;
- однаковий код має поводитися однаково у звичайному та ігровому режимі;
- не можна допускати зависання браузера через коротку програму.

## 2. Лексика

### 2.1. Коментарі

Коментар починається з `#` і триває до кінця рядка.

```ravlyk
# Це коментар
вперед 100 # Це теж коментар
```

### 2.2. Рядки

Рядки записуються в подвійних лапках. Поточний tokenizer підтримує рядки без переносу рядка всередині:

```ravlyk
клавіша "ArrowUp"
```

### 2.3. Дужки і розділювачі

Використовуються:

- `(` `)` для блоків, функцій і групування виразів;
- `,` для розділення аргументів;
- `=` для присвоєння і порівняння;
- `!=`, `<`, `>`, `<=`, `>=` для порівнянь.

### 2.4. Числа

Підтримуються числа такого типу:

```text
10
-10
3.14
.5
1e3
-2.5e-2
```

Числа мають бути скінченними. `Infinity`, `NaN` та інші JS-значення не є валідними значеннями мови.

## 3. Ідентифікатори

Ідентифікатори використовуються для змінних, функцій і параметрів.

Поточний код має `isValidParserIdentifier`, але потрібно формально зафіксувати правило.

Рекомендоване правило:

- дозволені українські й латинські літери;
- дозволені цифри після першого символу;
- дозволені `_` і `-`;
- назва не може бути порожньою;
- назва не може збігатися із зарезервованим словом;
- назва нормалізується до нижнього регістру.

## 4. Зарезервовані слова

Зарезервовані слова не можна використовувати як назви змінних, функцій або параметрів.

```text
вперед, назад, праворуч, ліворуч,
колір, фон, товщина, підняти, опустити,
очистити, перейти, повторити, повтори,
якщо, інакше, створити, грати,
край, клавіша, випадково,
forward, backward, right, left,
color, background, thickness, penup, pendown,
clear, goto, repeat, if, else, create, game,
edge, key, random
```

Поточна проблема: у коді вже є повідомлення для reserved-name помилок, але самі перевірки треба реалізувати в semantic validator.

## 5. Базові команди руху

### 5.1. Рух уперед

```ravlyk
вперед 100
forward 100
```

Аргумент — числовий вираз або `випадково` / `random`.

```ravlyk
вперед 50 + 50
вперед випадково
```

### 5.2. Рух назад

```ravlyk
назад 100
backward 100
```

### 5.3. Поворот праворуч

```ravlyk
праворуч 90
right 90
```

### 5.4. Поворот ліворуч

```ravlyk
ліворуч 90
left 90
```

## 6. Команди стану пера і полотна

### 6.1. Колір пера

```ravlyk
колір червоний
color red
колір випадково
```

Колір має бути в `COLOR_REGISTRY` з `constants.js` або його alias.

### 6.2. Колір фону

```ravlyk
фон білий
background white
фон випадково
```

### 6.3. Товщина лінії

```ravlyk
товщина 5
thickness 5
```

Поточні межі:

```js
MIN_PEN_SIZE = 1
MAX_PEN_SIZE = 50
```

Товщина має бути цілим числом.

### 6.4. Підняти / опустити олівець

```ravlyk
підняти
penup

опустити
pendown
```

### 6.5. Очистити полотно

```ravlyk
очистити
clear
```

### 6.6. Перейти в координату

```ravlyk
перейти 100 50
перейти в 100 50
перейти 100, 50
goto 100 50
goto to 100 50
перейти випадково
```

## 7. Вирази

Підтримуються:

- числові літерали;
- змінні;
- унарні `+` і `-`;
- бінарні `+`, `-`, `*`, `/`, `%`;
- групування дужками.

Приклади:

```ravlyk
створити a = 10
створити b = 20
вперед a + b * 2
вперед (a + b) * 2
вперед -a
```

Поточна проблема: треба явно протестувати ділення на нуль і нескінченні значення. Runtime має відхиляти нескінченний результат дружньою помилкою.

## 8. Змінні

### 8.1. Створення змінної

```ravlyk
створити x = 100
create x = 100
```

### 8.2. Присвоєння існуючій змінній

```ravlyk
x = 200
```

### Правила, які треба зафіксувати

- Змінну бажано створити через `створити` перед першим використанням.
- Не можна називати змінну зарезервованим словом.
- Не можна створювати змінну з назвою вже існуючої функції.
- Повторне `створити x = ...` для тієї самої змінної має або бути помилкою, або явно визначеним перевизначенням. Рекомендація: для дитячої мови краще помилка з підказкою використати `x = ...`.

## 9. Цикли

```ravlyk
повторити 4 (
  вперед 100
  праворуч 90
)
```

Alias:

```ravlyk
повтори 4 (...)
repeat 4 (...)
```

Кількість повторів має бути цілим числом `>= 0` і не більше `MAX_REPEATS_IN_LOOP`.

Поточний ліміт:

```js
MAX_REPEATS_IN_LOOP = 500
```

### Важливе уточнення

Ліміт на один цикл не захищає від вкладених циклів. Потрібен глобальний бюджет операцій.

## 10. Умови

### 10.1. Порівняння

```ravlyk
якщо x > 10 (
  колір червоний
)
```

Оператори:

```text
=, !=, <, >, <=, >=
```

### 10.2. Інакше

```ravlyk
якщо x > 10 (
  колір червоний
) інакше (
  колір синій
)
```

Alias:

```ravlyk
if x > 10 (...) else (...)
```

### 10.3. Умова краю

```ravlyk
якщо край (
  праворуч 180
)
```

Alias:

```ravlyk
if edge (...)
```

### 10.4. Умова клавіші

```ravlyk
якщо клавіша "ArrowUp" (
  вперед 10
)
```

Alias:

```ravlyk
if key "ArrowUp" (...)
```

## 11. Функції

### 11.1. Оголошення функції

```ravlyk
створити квадрат(сторона) (
  повторити 4 (
    вперед сторона
    праворуч 90
  )
)
```

### 11.2. Виклик функції

```ravlyk
квадрат(100)
```

### 11.3. Функція з кількома параметрами

```ravlyk
створити прямокутник(ширина, висота) (
  повторити 2 (
    вперед ширина
    праворуч 90
    вперед висота
    праворуч 90
  )
)

прямокутник(120, 60)
```

### Обов'язкові правила, які треба реалізувати

- Назва функції не може бути зарезервованим словом.
- Назва функції не може збігатися з назвою змінної.
- Не можна оголосити дві функції з однаковою назвою.
- Параметри не можуть дублюватися.
- Параметр не може бути зарезервованим словом.
- Кількість аргументів у виклику має точно збігатися з кількістю параметрів.
- Функція не має бути порожньою.

Поточна проблема: ці правила частково є в повідомленнях `constants.js`, але не реалізовані повністю.

## 12. Ігровий режим

Команда:

```ravlyk
грати (
  якщо клавіша "ArrowUp" (
    вперед 5
  )
)
```

Alias:

```ravlyk
game (...)
```

### Правила ігрового режиму

- У програмі має бути не більше одного блоку `грати`.
- `грати` має бути тільки на верхньому рівні.
- `грати` не можна вкладати у функції, цикли або умови.
- Якщо є `грати`, на верхньому рівні мають бути дозволені тільки:
  - створення змінних;
  - оголошення функцій;
  - сам блок `грати`.
- У кожному game tick має бути operation budget.

Поточний код уже має повідомлення:

- `GAME_MODE_TOP_LEVEL_ONLY`;
- `GAME_MODE_SINGLE_BLOCK`;
- `GAME_MODE_NESTED_BLOCK`.

Ці правила треба покрити тестами.

## 13. Випадковість

Поточні місця, де підтримується `випадково` / `random`:

```ravlyk
вперед випадково
колір випадково
фон випадково
перейти випадково
```

Правила:

- випадковий рух має не виводити Равлика за межі полотна;
- випадкова точка має бути в безпечній зоні;
- випадковий колір має бути з дозволеного registry;
- тести мають мати можливість підмінити random на deterministic seed або mock.

## 14. Помилки

Усі помилки мають бути дружніми й бажано містити:

- що сталося;
- де сталося: рядок/колонка;
- як виправити.

Технічні помилки типу `RangeError`, `TypeError`, `undefined is not a function` не мають показуватися дитині як основний текст.

## 15. Семантична модель виконання

Рекомендована модель:

1. `source code`;
2. tokenizer;
3. parser -> AST;
4. semantic validator -> checked AST;
5. runtime executes checked AST step by step;
6. renderer/animation layer відображає результат.

Не рекомендовано:

```text
AST -> попередньо розгорнута command queue -> виконання
```

Причина: передчасне обчислення значень змінних може відрізнятися від очікуваної послідовної семантики.

## 16. Мінімальна граматика

Умовна EBNF-форма:

```ebnf
program        = statement* ;
statement      = move | turn | color | background | thickness | pen | clear | goto
               | assignment | createVar | functionDef | functionCall
               | repeat | ifStmt | gameStmt ;

move           = ("вперед" | "forward" | "назад" | "backward") (expression | random) ;
turn           = ("праворуч" | "right" | "ліворуч" | "left") expression ;
color          = ("колір" | "color") (identifier | random) ;
background     = ("фон" | "background") (identifier | random) ;
thickness      = ("товщина" | "thickness") unsignedInteger ;
pen            = "підняти" | "penup" | "опустити" | "pendown" ;
clear          = "очистити" | "clear" ;
goto           = ("перейти" | "goto") ["в" | "to"] ((expression [","] expression) | random) ;

createVar      = ("створити" | "create") identifier "=" expression ;
assignment     = identifier "=" expression ;
functionDef    = ("створити" | "create") identifier "(" [params] ")" block ;
functionCall   = identifier "(" [args] ")" ;
params         = identifier ("," identifier)* ;
args           = expression ("," expression)* ;

repeat         = ("повторити" | "повтори" | "repeat") expression block ;
ifStmt         = ("якщо" | "if") condition block [("інакше" | "else") block] ;
gameStmt       = ("грати" | "game") block ;
block          = "(" statement* ")" ;

condition      = "край" | "edge"
               | ("клавіша" | "key") string
               | expression comparisonOperator expression ;
comparisonOperator = "=" | "!=" | "<" | ">" | "<=" | ">=" ;

expression     = unaryExpr (("+" | "-" | "*" | "/" | "%") unaryExpr)* ;
unaryExpr      = ["+" | "-"] primary ;
primary        = number | identifier | "(" expression ")" ;
random         = "випадково" | "random" ;
```

## 17. Тести відповідності специфікації

Кожна команда має мати тести:

- parser acceptance;
- parser rejection;
- semantic validation;
- runtime behavior;
- friendly error message.

Рекомендований файл:

```text
tests/unit/language-spec.test.js
```

## 18. Відомі розбіжності між специфікацією і кодом

1. Reserved names описані в повідомленнях, але не перевіряються повністю.
2. Дублікати параметрів функції не заборонені.
3. Зайві аргументи функції можуть ігноруватися.
4. Вкладені цикли можуть створювати надто багато команд.
5. Звичайний runtime і game runtime мають різні шляхи виконання.
6. Немає єдиного semantic validation етапу.
7. Немає формального operation budget.

## 19. Definition of Done для мови

Мову можна вважати стабільною, коли:

- `LANGUAGE_SPEC.md` відповідає реалізації;
- усі правила covered тестами;
- звичайний режим і game mode мають однакову семантику;
- усі помилки мають дружні повідомлення;
- неможливо зависити браузер короткою програмою;
- додавання нової команди вимагає оновлення spec, parser, runtime, tests і manual.
