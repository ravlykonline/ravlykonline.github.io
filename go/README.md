# Пригоди Равлика

**Пригоди Равлика** — статична навчальна гра з основ алгоритмічного мислення для дітей. Гравець допомагає равлику дістатися яблука, розкладаючи стрілки на полі.

Публічна адреса гри:

```text
https://ravlyk.org/go
```

Проєкт має залишатися простим: одна окрема гра, 20 відкритих рівнів, GitHub Pages, без реєстрації, без бекенду, без аналітики й без довготривалого збереження прогресу.

---

## Ключові рішення

- **Тип проєкту:** статичний сайт / PWA.
- **Хостинг:** GitHub Pages.
- **Адреса:** `ravlyk.org/go`.
- **Бекенд:** відсутній.
- **Реєстрація:** відсутня.
- **Cookies:** не використовуються.
- **Аналітика:** не використовується.
- **Зовнішні CDN:** не використовуються.
- **Кількість рівнів:** рівно 20.
- **Доступність рівнів:** усі 20 рівнів відкриті одразу.
- **Збереження прогресу:** тільки `sessionStorage`, тільки в межах відкритої вкладки.
- **Після закриття вкладки:** усе починається з нуля.
- **PWA:** гра має працювати офлайн після першого відкриття.
- **Рефакторинг:** м'який, без суттєвої зміни зовнішнього вигляду.

---

## Цільова аудиторія

Гра призначена для:

- учнів початкової школи;
- дітей, які тільки знайомляться з алгоритмами;
- вчителів інформатики та початкових класів;
- використання на інтерактивній дошці, планшеті, ноутбуці, ПК і телефоні.

Інтерфейс має бути зрозумілим без довгих інструкцій. Дитина має швидко зрозуміти: треба поставити стрілки, запустити равлика й перевірити маршрут.

---

## Мета рефакторингу

Мета рефакторингу — не створити платформу й не переписати гру у фреймворку. Мета — зробити наявну гру чистішою, безпечнішою та простішою для підтримки.

Після рефакторингу гра має:

1. виглядати майже так само;
2. працювати стабільніше;
3. використовувати ES-модулі замість глобального `window.SnailGame`;
4. мати чітке розділення логіки гри, UI, стану, рівнів, звуку й PWA;
5. зберігати поточний рівень, розставлені стрілки та пройдені рівні тільки в `sessionStorage`;
6. мати кнопку **«Почати гру спочатку»**;
7. працювати офлайн після першого відкриття;
8. мати тести, які захищають ключові архітектурні правила.

## Запуск локально

Рекомендовано запускати через локальний HTTP-сервер, а не через `file://`, особливо для перевірки PWA/service worker.

```bash
python3 -m http.server 8080
```

Потім відкрити:

```text
http://localhost:8080/
```

---

## Тести

Поточний запуск тестів:

```bash
npm test
```

Тести мають залишатися простими й запускатися без build-процесу.

Після рефакторингу тести мають перевіряти:

- у проєкті є рівно 20 рівнів;
- усі рівні мають унікальні `id`;
- усі рівні валідні;
- кожен рівень має розв'язок;
- немає зовнішніх скриптів і CDN;
- немає `eval`, `new Function`, `document.write`;
- `localStorage` не використовується;
- `sessionStorage` використовується тільки через `js/state/sessionStore.js`;
- PWA-кеш містить реальні файли;
- `manifest.json` коректний для `/go`;
- текстові файли не мають BOM.

## Розгортання на GitHub Pages

Проєкт має працювати за адресою:

```text
https://ravlyk.org/go
```

Тому всі внутрішні шляхи мають бути відносними:

```html
<link rel="manifest" href="./manifest.json">
<script type="module" src="./js/main.js"></script>
```

Не використовувати абсолютні шляхи:

```text
/js/main.js
/css/game.css
/assets/snail.svg
/sw.js
```

Для `/go` такі шляхи можуть вказувати не на теку гри, а на корінь домену.

## Що не можна додавати без окремого рішення

Не додавати:

- React, Vue, Svelte або інший фреймворк;
- Vite, Webpack або інший build-процес;
- бекенд;
- реєстрацію;
- cookies;
- Google Analytics, Plausible або іншу аналітику;
- зовнішні CDN;
- `localStorage` для прогресу;
- збір персональних даних;
- форми з email, ім'ям, телефоном або іншими даними дитини.

---

## Політика UX під час рефакторингу

Рефакторинг не має змінювати гру для дитини. Дозволені тільки обережні виправлення:

- виправлення очевидних помилок;
- покращення доступності;
- стабілізація мобільної версії;
- покращення роботи клавіатури;
- додавання кнопки **«Почати гру спочатку»**;
- уточнення текстів, якщо це не змінює сценарій гри.

Не змінювати без окремого рішення:

- загальний стиль;
- композицію основного екрана;
- механіку розкладання стрілок;
- карту рівнів;
- логіку відкритості всіх рівнів;
- кількість рівнів.
