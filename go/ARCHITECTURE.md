# ARCHITECTURE — Пригоди Равлика

Цей документ описує цільову архітектуру гри **«Пригоди Равлика»** після м'якого рефакторингу.

Головна ідея: зберегти гру простою, статичною і незалежною від фреймворків, але прибрати крихку глобальну архітектуру та розділити код на зрозумілі ES-модулі.

---

## Архітектурні межі

Проєкт є однією окремою грою за адресою:

```text
ravlyk.org/go
```

Проєкт не є платформою, CMS, кабінетом учня або системою керування навчанням.

У межах цього проєкту не передбачено:

- бекенд;
- база даних;
- реєстрація;
- авторизація;
- ролі користувачів;
- кабінет учня;
- кабінет вчителя;
- серверне збереження прогресу;
- аналітика;
- cookies;
- сторонні CDN.

---

## Архітектурний стиль

Цільовий стиль:

```text
Static-first vanilla JavaScript application
ES Modules
No build step
No framework
No backend
No external runtime dependencies
PWA/offline-capable
```

Проєкт має запускатися на GitHub Pages без збірки. Усі файли, які потрібні браузеру, мають лежати в репозиторії як звичайні HTML/CSS/JS/assets.

---

## Чому не потрібен фреймворк

Для цієї гри React/Vue/Svelte/Vite не дають достатньої користі, але додають:

- складніший tooling;
- залежності;
- ризики supply chain;
- потребу в build-процесі;
- складніший деплой;
- більший обсяг коду;
- вищий поріг входу для майбутнього редагування.

Для 20-рівневої статичної гри достатньо нативних ES-модулів.

---

## Поточна проблема

Поточна версія використовує глобальний об'єкт:

```js
window.SnailGame
```

Різні JS-файли послідовно додають у нього властивості й функції. Це працює, але створює проблеми:

- порядок `<script>` у `index.html` критично важливий;
- залежності між файлами неявні;
- складно тестувати окремі частини;
- один файл може випадково перезаписати інший;
- код стає важко підтримувати при збільшенні кількості файлів.

Після рефакторингу має бути один вхідний файл:

```html
<script type="module" src="./js/main.js"></script>
```

А всередині `main.js` — явні імпорти.

## Ролі модулів

### `js/main.js`

Вхідна точка застосунку.

Відповідає за:

- імпорт модулів;
- створення початкового стану;
- підключення UI-подій;
- первинний рендер;
- запуск PWA-реєстрації;
- координацію між state, engine і UI.

Не має містити великих шматків логіки гри або DOM-рендерингу.

---

### `js/core/config.js`

Глобальні налаштування гри:

```js
export const GAME_CONFIG = {
  totalLevels: 20,
  storageKey: 'ravlyk-code-session-v1',
  stepMs: 540,
  maxSteps: 80,
  routeScanOrder: ['down', 'right', 'up', 'left']
};
```

Тут мають бути тільки стабільні конфігураційні значення.

---

### `js/core/constants.js`

Сталі значення домену:

- напрямки;
- типи клітинок;
- типи рівнів;
- коди плиток/стрілок;
- відповідність стрілки й asset-файлу.

Приклад:

```js
export const DIRECTIONS = ['up', 'right', 'down', 'left'];
export const LEVEL_TYPES = ['play', 'debug'];
```

---

### `js/core/levels.js`

Опис 20 рівнів.

На цьому етапі рівні можуть залишатися JS-модулем, а не JSON, бо це простіше для статичного сайту без build-процесу.

Файл має експортувати:

```js
export const levels = [...];
```

Не має створювати глобальні змінні й не має звертатися до DOM.

---

### `js/core/texts.uk.js`

Українські тексти інтерфейсу:

```js
export const textsUk = {
  run: 'Запустити',
  clear: 'Очистити',
  restartGame: 'Почати гру спочатку'
};
```

Тексти конкретних рівнів можуть лишатися в `levels.js`, якщо так зручніше для педагогічного редагування. Загальні UI-тексти — у `texts.uk.js`.

---

### `js/engine/simulator.js`

Чиста логіка проходження рівня.

Має приймати дані:

```js
simulateLevel({ level, arrows, config })
```

І повертати результат:

```js
{
  success: true,
  path: [
    { r: 2, c: 2, facing: 'right' },
    { r: 2, c: 3, facing: 'right' }
  ],
  reason: null
}
```

Або при помилці:

```js
{
  success: false,
  path: [...],
  reason: 'missing-arrow' | 'blocked' | 'out-of-board' | 'loop' | 'wrong-turn'
}
```

Engine не має:

- читати DOM;
- писати в DOM;
- запускати анімацію;
- грати звук;
- читати або писати `sessionStorage`;
- знати про кнопки, модалки чи CSS-класи.

---

### `js/engine/validation.js`

Валідація рівнів і ходів.

Має перевіряти:

- рівно 20 рівнів;
- унікальні `id`;
- коректні `rows` і `cols`;
- старт у межах поля;
- яблуко в межах поля;
- перешкоди в межах поля;
- старт не збігається з яблуком;
- старт/яблуко не збігаються з перешкодами;
- усі `allowedTiles` існують;
- усі `presetArrows` у межах поля;
- усі `presetArrows` дозволені або явно позначені як виняток для debug-рівня.

---

### `js/engine/levelRules.js`

Правила домену:

- чи можна ставити стрілку в клітинку;
- чи є клітинка перешкодою;
- чи є клітинка стартом;
- чи є клітинка яблуком;
- чи дозволена стрілка на поточному рівні;
- як інтерпретувати поворотну стрілку.

Цей файл також не має знати про DOM.

---

### `js/state/sessionStore.js`

Єдине місце, де дозволено працювати з `sessionStorage`.

Має експортувати:

```js
loadSession()
saveSession(data)
clearSession()
isSessionStorageAvailable()
```

Файл має бути defensive:

- якщо `sessionStorage` недоступний, гра все одно працює в пам'яті;
- якщо JSON пошкоджений, він ігнорується;
- якщо схема збереження стара або неправильна, вона ігнорується;
- помилки storage не мають ламати гру.

`localStorage` у цьому проєкті не використовується.

---

### `js/state/gameState.js`

Створення й оновлення стану гри.

Стан має містити:

```js
{
  currentLevelId: 1,
  arrowsByLevel: {
    "1": {
      "2,3": "right"
    }
  },
  completedLevelIds: [1, 2, 3],
  selectedTile: null,
  running: false,
  dragState: null
}
```

Важливо: за вимогою продукту поточні стрілки мають зберігатися через `sessionStorage` у межах вкладки. Тому `arrowsByLevel` має бути частиною session payload.

Після закриття вкладки все зникає автоматично, бо використовується `sessionStorage`.

---

### `js/ui/dom.js`

Єдине місце для пошуку DOM-елементів:

```js
export function getDomRefs() {
  return {
    gridEl: document.getElementById('grid'),
    btnRun: document.getElementById('btn-run')
  };
}
```

Інші UI-модулі мають отримувати refs явно, а не робити хаотичні `document.getElementById`.

---

### `js/ui/renderBoard.js`

Рендер поля гри:

- клітинки;
- равлик;
- яблуко;
- перешкоди;
- стрілки;
- доступні/недоступні клітинки;
- keyboard/touch атрибути.

Не має містити логіку проходження маршруту.

---

### `js/ui/renderPalette.js`

Рендер палітри стрілок для поточного рівня.

Відповідає за:

- доступні стрілки;
- активний вибір;
- drag/touch поведінку;
- aria-label для кнопок/елементів.

---

### `js/ui/renderLevelMap.js`

Рендер карти 20 рівнів.

Правила:

- усі 20 рівнів відкриті одразу;
- поточний рівень виділений;
- пройдені рівні підсвічуються тільки в межах поточної сесії;
- карта не блокує перехід до непройдених рівнів.

---

### `js/ui/renderProgress.js`

Оновлення прогресу:

- текст прогресу;
- заповнення progress bar;
- ARIA-атрибути.

Важливо: `aria-valuenow` має оновлюватися саме на елементі з `role="progressbar"`.

---

### `js/ui/modals.js`

Усі модальні вікна:

- підказка/інформація про рівень;
- карта рівнів;
- підтвердження очищення рівня;
- підтвердження **«Почати гру спочатку»**.

У destructive-діях мають бути явні кнопки:

```text
Скасувати
Почати спочатку
```

або

```text
Скасувати
Очистити
```

Закриття через `×` не замінює явну кнопку «Скасувати».

---

### `js/ui/focus.js`

Логіка доступності й фокусу:

- focus trap у модалках;
- повернення фокусу на кнопку, яка відкрила модалку;
- keyboard navigation на полі;
- Escape для закриття модалки;
- Enter/Space для активації.

---

### `js/features/audio.js`

Звукові ефекти.

Правила:

- звук не має бути критично потрібним для проходження гри;
- гра має працювати без Audio API;
- помилки Audio API не мають ламати гру;
- звук має запускатися тільки після дії користувача, щоб браузер не блокував його.

---

### `js/features/speech.js`

Озвучення завдання через Web Speech API.

Правила:

- якщо API недоступний, кнопка має бути прихована або неактивна;
- озвучення не має бути єдиним способом отримати інструкцію;
- текст завдання завжди має бути видимим.

---

### `js/features/confetti.js`

Візуальне святкування успіху.

Правила:

- має поважати `prefers-reduced-motion`;
- не має блокувати гру;
- не має містити hardcoded inline styles, якщо їх можна замінити CSS-класами;
- не має використовувати сторонні бібліотеки.

---

### `js/features/pwaRegister.js`

Реєстрація service worker:

```js
export function registerPwa() {
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.register('./sw.js');
  }
}
```

Для GitHub Pages і `/go` важливо використовувати відносний шлях `./sw.js`.

---

## Потік даних

Бажаний напрямок залежностей:

```text
levels/texts/config
        ↓
state ← main → ui
        ↓      ↑
      engine ──┘
```

UI викликає дії, state оновлюється, engine рахує результат, UI показує результат.

Небажані залежності:

- engine → DOM;
- engine → storage;
- levels → DOM;
- sessionStore → UI;
- UI → прямі мутації storage;
- випадкові глобальні змінні.

---

## Session payload

Цільова структура збереження:

```js
{
  version: 1,
  currentLevelId: 7,
  arrowsByLevel: {
    "7": {
      "1,2": "right",
      "1,3": "down"
    }
  },
  completedLevelIds: [1, 2, 3, 6]
}
```

Поведінка:

- оновлення сторінки зберігає поточний рівень, стрілки й пройдені рівні;
- перехід між рівнями зберігає стрілки кожного рівня в межах сесії;
- закриття вкладки очищує все автоматично;
- кнопка **«Почати гру спочатку»** очищує session payload і повертає на рівень 1.

## PWA й шляхи

Оскільки гра живе в `/go`, усі шляхи мають бути відносними:

```html
<link rel="manifest" href="./manifest.json">
<script type="module" src="./js/main.js"></script>
```

У service worker:

```js
const APP_SHELL = [
  './',
  './index.html',
  './offline.html',
  './manifest.json',
  './css/tokens.css',
  './css/base.css',
  './css/game.css',
  './js/main.js'
];
```

Не використовувати абсолютні шляхи `/js/...`, `/css/...`, `/assets/...`.

---

## Архітектурні заборони

Не використовувати:

```js
eval(...)
new Function(...)
document.write(...)
localStorage
```

Не додавати:

- CDN;
- external scripts;
- remote fonts;
- analytics;
- cookies;
- backend API;
- build step;
- framework.

`innerHTML` бажано прибрати повністю. Для очищення DOM використовувати:

```js
element.replaceChildren();
```

Для створення елементів використовувати:

```js
document.createElement(...)
```

---

## Принципи рефакторингу

1. Один логічний крок — один невеликий коміт.
2. Після кожного кроку запускати `npm test`.
3. Не змінювати UX без потреби.
4. Не змішувати рефакторинг з редизайном.
5. Не додавати залежності.
6. Не змінювати кількість рівнів.
7. Не ламати PWA/offline.
8. Не перетворювати гру на платформу.

---

## Критерії готовності

Рефакторинг можна вважати завершеним, якщо:

- `index.html` підключає тільки `./js/main.js` як `type="module"`;
- немає `window.SnailGame` як application state;
- engine не залежить від DOM;
- `sessionStorage` використовується тільки через `sessionStore.js`;
- `localStorage` не використовується;
- 20 рівнів доступні одразу;
- поточний рівень, стрілки й пройдені рівні зберігаються після reload у межах вкладки;
- після закриття вкладки прогрес не зберігається;
- є кнопка **«Почати гру спочатку»**;
- PWA працює на `/go`;
- тести проходять;
- зовнішній вигляд гри істотно не змінився.
