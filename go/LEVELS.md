# LEVELS — Пригоди Равлика

Цей документ описує формат рівнів, правила їх редагування й перевірки.

У грі має бути **рівно 20 рівнів**. Усі рівні відкриті одразу. Підсвітка на карті рівнів показує лише те, що дитина вже пройшла в межах поточної вкладки.

---

## Основні правила

1. Рівнів має бути рівно 20.
2. Усі `id` мають бути унікальними.
3. `id` бажано тримати послідовними: `1`–`20`.
4. Усі рівні відкриті з першого запуску.
5. Проходження одного рівня не блокує і не відкриває інші.
6. Пройдені рівні зберігаються тільки в `sessionStorage`.
7. Після закриття вкладки всі позначки проходження зникають.
8. Кожен рівень має бути педагогічно осмисленим, а не просто складним.

---

## Поточний формат рівня

Поточна структура рівня близька до такої:

```js
{
  id: 1,
  name: 'Пряма доріжка',
  type: 'play',
  rows: 6,
  cols: 8,
  start: { r: 2, c: 2 },
  apple: { r: 2, c: 5 },
  obstacles: [],
  startFacing: 'right',
  allowedTiles: ['right'],
  presetArrows: {},
  hint: 'Перетягни стрілочку праворуч на клітинки між равликом і яблуком.',
  goal: 'Навчись рухатися праворуч.'
}
```

Після переходу на ES-модулі файл має експортувати рівні так:

```js
export const levels = [
  {
    id: 1,
    name: 'Пряма доріжка',
    type: 'play',
    rows: 6,
    cols: 8,
    start: { r: 2, c: 2 },
    apple: { r: 2, c: 5 },
    obstacles: [],
    startFacing: 'right',
    allowedTiles: ['right'],
    presetArrows: {},
    hint: '...',
    goal: '...'
  }
];
```

---

## Поля рівня

### `id`

Числовий ідентифікатор рівня.

Вимоги:

- число;
- унікальне;
- від 1 до 20;
- не змінювати без потреби, бо `id` використовується в session state.

---

### `name`

Назва рівня, яку бачить користувач.

Вимоги:

- українською мовою;
- коротка;
- зрозуміла дитині;
- бажано до 3–5 слів;
- без технічних термінів, якщо вони не пояснені в контексті.

Добре:

```text
Пряма доріжка
Поворот до яблука
Обходимо перешкоди
```

Гірше:

```text
Оптимізація маршруту з умовною зміною напрямку
```

---

### `type`

Тип рівня.

Допустимі значення:

```js
'play'
'debug'
```

`play` — дитина будує маршрут сама.

`debug` — маршрут частково заданий, але має помилки, які потрібно виправити.

---

### `rows` і `cols`

Розмір поля.

Поточний стандарт:

```js
rows: 6,
cols: 8
```

Для м'якого рефакторингу бажано не змінювати розміри без потреби, щоб не зламати UX на мобільних і планшетах.

---

### `start`

Позиція равлика на старті:

```js
start: { r: 2, c: 2 }
```

`r` — рядок, `c` — колонка. Індексація з нуля.

Вимоги:

- позиція в межах поля;
- не збігається з яблуком;
- не збігається з перешкодою;
- не має бути недосяжною для першої дії.

---

### `apple`

Позиція яблука:

```js
apple: { r: 2, c: 5 }
```

Вимоги:

- позиція в межах поля;
- не збігається зі стартом;
- не збігається з перешкодою;
- має бути досяжною правильним маршрутом.

---

### `obstacles`

Перешкоди на полі:

```js
obstacles: [
  { r: 3, c: 2, kind: 'rock', label: 'Каміння' },
  { r: 3, c: 3, kind: 'log', label: 'Колода' }
]
```

Допустимі `kind` залежать від наявних assets. Поточно використовуються:

```js
'rock'
'log'
```

Правила:

- перешкоди не мають збігатися між собою;
- не мають стояти на старті;
- не мають стояти на яблуку;
- не мають повністю блокувати всі можливі маршрути, якщо це не навчально виправданий debug-рівень.

---

### `startFacing`

Початковий напрямок равлика:

```js
startFacing: 'right'
```

Допустимі значення:

```js
'up'
'right'
'down'
'left'
```

Якщо поле відсутнє, стандартне значення — `right`.

---

### `allowedTiles`

Список стрілок, доступних на рівні:

```js
allowedTiles: ['up', 'right', 'down', 'left']
```

Це важливий педагогічний інструмент. Не треба відкривати всі стрілки на ранніх рівнях, якщо це збільшує когнітивне навантаження.

---

## Напрямки й стрілки

Базові прямі стрілки:

```js
'up'
'right'
'down'
'left'
```

Поворотні стрілки:

```js
'right-up'
'down-right'
'left-down'
'up-left'
'right-down'
'down-left'
'left-up'
'up-right'
```

Поворотна стрілка кодує вхідну сторону й напрямок виходу. Її інтерпретація має залишатися відповідальністю engine, а не UI.

---

### `presetArrows`

Заздалегідь поставлені стрілки:

```js
presetArrows: {
  '3,2': 'right',
  '3,3': 'right',
  '3,4': 'left'
}
```

Ключ — координата у форматі:

```text
r,c
```

Значення — код стрілки.

Використовується особливо для debug-рівнів.

Правила:

- координата має бути в межах поля;
- не ставити стрілки на старт і яблуко, якщо engine/UI цього не підтримують явно;
- не ставити стрілки на перешкоди;
- стрілки мають входити до дозволених або бути свідомим винятком для debug-рівня.

---

### `hint`

Підказка для дитини.

Вимоги:

- коротка;
- дружня;
- без звинувачень;
- не розкриває повністю рішення, якщо рівень має бути самостійним;
- пояснює наступну корисну дію.

Добре:

```text
Спробуй обійти каміння зверху.
```

Гірше:

```text
Постав right у 2,3, потім down у 2,4, потім left у 3,4.
```

---

### `goal`

Педагогічна мета рівня.

Це текст для пояснення, що саме тренує рівень.

Приклади:

```text
Навчись рухатися праворуч.
Навчись перевіряти і виправляти послідовність стрілок.
Навчись будувати складніший шлях між перешкодами.
```

---

## Педагогічна прогресія

Рівні мають поступово вводити нові ідеї:

1. рух праворуч;
2. рух угору;
3. рух ліворуч;
4. рух униз;
5. виправлення простого неправильного маршруту;
6. перший поворот;
7. звивистий шлях;
8. перешкоди;
9. обхід перешкод;
10. довший маршрут;
11. debug з поворотами;
12–20. складніші маршрути з усіма або майже всіма стрілками.

Під час педагогічного редагування важливо дивитися не тільки на складність, а й на те, чи зрозуміло дитині, що саме вона зараз вивчає.

---

## Правила складності

Для молодших школярів складність має зростати поступово.

Не робити різких стрибків:

- з 1–2 доступних стрілок одразу до 12;
- з прямого маршруту одразу до багатьох поворотів;
- з відкритого поля одразу до лабіринту;
- з play-рівня одразу до складного debug-рівня.

Краще:

```text
новий тип стрілки → простий рівень на цей тип → рівень із перешкодою → рівень на комбінування
```

---

## Валідація рівнів

Після рефакторингу бажано мати функцію:

```js
validateLevels(levels)
```

Вона має перевіряти:

- рівно 20 рівнів;
- усі `id` унікальні;
- кожен рівень має `name`, `type`, `rows`, `cols`, `start`, `apple`, `allowedTiles`, `hint`, `goal`;
- `type` тільки `play` або `debug`;
- `rows` і `cols` додатні;
- координати в межах поля;
- немає дубльованих перешкод;
- немає перешкод на старті/яблуку;
- усі стрілки існують;
- усі `presetArrows` коректні;
- рівень має хоча б один розв'язок.

---

## Автоматична перевірка розв'язності

Кожен рівень має бути розв'язним.

Тести мають проходити:

```bash
npm test
```

Якщо рівень педагогічно задуманий як debug-рівень з помилками, він усе одно має мати можливість бути виправленим доступними стрілками.

---

## Редагування текстів рівнів

Під час редагування назв, підказок і цілей:

- писати українською;
- уникати канцеляриту;
- уникати надто довгих речень;
- звертатися до дитини дружньо;
- не використовувати слова «помилка» як звинувачення;
- краще писати «спробуй», «допоможи», «перевір», ніж «ти неправильно зробив».

---

## Checklist після зміни рівня

Після кожної зміни рівня перевірити:

- [ ] `npm test` проходить.
- [ ] Рівнів усе ще 20.
- [ ] `id` не дублюються.
- [ ] Старт і яблуко в межах поля.
- [ ] Перешкоди не перекривають старт/яблуко.
- [ ] Доступні стрілки відповідають педагогічній меті.
- [ ] Підказка зрозуміла дитині.
- [ ] Рівень можна пройти.
- [ ] Рівень нормально виглядає на телефоні.
- [ ] Рівень нормально виглядає на інтерактивній дошці.

---

## Що не робити

Не додавати:

- більше 20 рівнів без окремого рішення;
- блокування рівнів;
- реєстрацію для збереження прогресу;
- локальний постійний прогрес у `localStorage`;
- складний редактор рівнів у межах цієї гри;
- JSON-завантаження з віддаленого сервера;
- рівні, які вимагають читати довгі інструкції.

---

## Можливий майбутній формат JSON

Якщо колись буде потрібно, рівні можна винести в JSON:

```text
data/levels.uk.json
```

Але для поточного етапу це не обов'язково. ES-модуль `levels.js` є достатнім і простішим рішенням для GitHub Pages.
