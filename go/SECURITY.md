# SECURITY — Пригоди Равлика

Цей документ описує безпекову модель гри **«Пригоди Равлика»**.

Головний принцип: гра має бути безпечною завдяки простоті. Вона не повинна збирати персональні дані, не повинна мати бекенду, не повинна використовувати сторонні скрипти й не повинна зберігати прогрес після закриття вкладки.

---

## Коротка модель безпеки

Проєкт є статичною грою за адресою:

```text
ravlyk.org/go
```

У грі немає:

- реєстрації;
- авторизації;
- особистого кабінету;
- бази даних;
- бекенду;
- cookies;
- аналітики;
- рекламних скриптів;
- сторонніх CDN;
- форм для введення персональних даних;
- серверного збереження прогресу.

---

## Дані користувача

Гра не збирає персональні дані.

Гра не просить і не зберігає:

- ім'я дитини;
- email;
- телефон;
- школу;
- клас;
- IP-адресу на рівні застосунку;
- результати проходження на сервері;
- поведінкову аналітику;
- cookies.

---

## SessionStorage

Гра використовує тільки `sessionStorage`.

Цільове призначення:

- зберегти поточний рівень після оновлення сторінки;
- зберегти розкладені стрілки після оновлення сторінки;
- підсвітити пройдені рівні в межах відкритої вкладки.

Після закриття вкладки прогрес має зникати автоматично.

Це важливо для шкільних комп'ютерів і спільних пристроїв: наступна дитина не має бачити довготривалий прогрес попередньої дитини.

---

## Заборонено використовувати localStorage

`localStorage` не використовується в цьому проєкті.

Причина: `localStorage` зберігається між сесіями й може показати прогрес попередньої дитини на спільному пристрої.

Тести мають перевіряти, що `localStorage` не використовується в коді.

Єдиний дозволений storage API:

```js
window.sessionStorage
```

І тільки через файл:

```text
js/state/sessionStore.js
```

---

## Session payload

Дозволена структура збереження:

```js
{
  version: 1,
  currentLevelId: 7,
  arrowsByLevel: {
    "7": {
      "1,2": "right"
    }
  },
  completedLevelIds: [1, 2, 3]
}
```

Не зберігати:

- ім'я;
- email;
- клас;
- школу;
- дату народження;
- довільний текст, введений користувачем;
- будь-які ідентифікатори дитини.

---

## Зовнішні ресурси

Заборонено додавати:

```html
<script src="https://...">
<link href="https://...">
<img src="https://...">
```

Без окремого рішення не використовувати:

- CDN;
- Google Fonts;
- Font Awesome CDN;
- Google Analytics;
- Plausible;
- Sentry;
- Hotjar;
- YouTube embeds;
- remote images;
- remote audio;
- remote JSON.

Усі ресурси мають бути локальними файлами проєкту.

---

## Заборонені небезпечні API

Не використовувати:

```js
eval(...)
new Function(...)
document.write(...)
setTimeout("code as string")
setInterval("code as string")
```

Причина: ці API виконують рядки як код або небезпечно змінюють документ.

---

## innerHTML

Бажана ціль — повністю прибрати `innerHTML`.

Для очищення елемента використовувати:

```js
element.replaceChildren();
```

Для створення DOM:

```js
const img = document.createElement('img');
img.src = './assets/snail.svg';
img.alt = 'Равлик';
```

Якщо `innerHTML` тимчасово лишається під час проміжного рефакторингу, він має бути в whitelist тестів і не має приймати користувацький текст.

---

## XSS-модель

Зараз гра не має полів довільного введення користувача, тому XSS-ризик низький.

Але під час рефакторингу не можна додавати механізми, які вставляють у DOM неперевірений HTML.

Правило:

- текст — через `textContent`;
- атрибути — через `setAttribute` або властивості DOM;
- елементи — через `createElement`;
- HTML-рядки — уникати.

---

## GitHub Pages і security headers

GitHub Pages не дає такого ж прямого контролю над HTTP security headers, як Cloudflare Pages.

Тому захист має забезпечуватися насамперед архітектурою:

- немає сторонніх скриптів;
- немає довільного HTML;
- немає форм персональних даних;
- немає бекенду;
- немає cookies;
- немає аналітики;
- немає `localStorage`;
- мінімізовано `innerHTML`.

Якщо проєкт колись буде перенесено на Cloudflare Pages або власний сервер, бажано додати такі headers:

```text
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'; img-src 'self' data:; manifest-src 'self'; worker-src 'self'; connect-src 'self'; form-action 'none'; upgrade-insecure-requests
X-Content-Type-Options: nosniff
Referrer-Policy: no-referrer
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=(), bluetooth=(), magnetometer=(), gyroscope=(), accelerometer=()
X-Frame-Options: DENY
```

Для GitHub Pages ці headers не треба додавати в `_headers`, бо GitHub Pages їх не застосує. Але код має бути готовий до майбутнього суворого CSP.

## Service worker

Service worker має працювати тільки в межах цієї гри.

Реєстрація:

```js
navigator.serviceWorker.register('./sw.js');
```

Не використовувати абсолютний шлях:

```js
navigator.serviceWorker.register('/sw.js');
```

Для `ravlyk.org/go` це може зареєструвати service worker не там, де треба.

Service worker не має:

- звертатися до сторонніх доменів;
- кешувати сторонні ресурси;
- перехоплювати non-GET запити;
- працювати як проксі для зовнішніх сайтів.

## Privacy text для сторінки або футера

Якщо потрібен короткий текст для користувача:

```text
Гра не має реєстрації та не збирає персональні дані. Поточний рівень і стрілки можуть тимчасово зберігатися лише в цій вкладці браузера, щоб гра не зникала після оновлення сторінки. Після закриття вкладки ці дані видаляються браузером.
```

## Security checklist перед релізом

Перед публікацією перевірити:

- [ ] `npm test` проходить без помилок.
- [ ] У коді немає `localStorage`.
- [ ] У коді немає `eval` і `new Function`.
- [ ] У коді немає сторонніх URL.
- [ ] У `index.html` немає CDN.
- [ ] У `manifest.json` немає зовнішніх іконок.
- [ ] У `sw.js` кешуються тільки локальні файли.
- [ ] `sessionStorage` використовується тільки через `sessionStore.js`.
- [ ] Після закриття вкладки прогрес не зберігається.
- [ ] Кнопка **«Почати гру спочатку»** очищує session payload.
- [ ] Гра працює без інтернету після першого відкриття.
- [ ] На спільному пристрої не лишається довготривалий прогрес учня.

---

## Що робити при виявленні проблеми

Якщо знайдено проблему безпеки:

1. Не додавати тимчасовий небезпечний workaround.
2. Зафіксувати сценарій відтворення.
3. Додати тест, який ловить проблему.
4. Виправити мінімальною зміною.
5. Перевірити, що UX не змінився без потреби.
6. Оновити цей документ, якщо правило безпеки стало точнішим.
