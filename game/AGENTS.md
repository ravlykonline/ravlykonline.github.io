# AGENTS.md — інструкції для Codex, Claude Code та інших агентів

Цей файл є головною інструкцією для будь-якого AI-агента, який змінює код проєкту «Равлик-бродилка».

Перед змінами агент має прочитати цей файл повністю.

---

## 1. Контекст проєкту

Це проста браузерна навчальна гра для дітей.

Дитина керує Равликом, збирає яблука, зустрічає звірів, виконує короткі завдання й отримує зірочки.

Проєкт на ранньому етапі, тому дозволені фундаментальні зміни, але тільки якщо вони:

- спрощують підтримку;
- підсилюють безпеку;
- покращують доступність;
- не збільшують когнітивне навантаження для дитини;
- не руйнують простий static-site підхід.

## 3. Найважливіші незмінні правила

### 3.1 Не зберігати прогрес між сесіями

Заборонено зберігати між reload/open:

- яблука;
- зірочки;
- виконані NPC;
- відповіді;
- результати;
- імена дітей;
- індивідуальний прогрес.

Причина: спільні шкільні комп'ютери.

### 3.2 Не додавати персональні дані

Не додавати:

- логін;
- реєстрацію;
- ім'я дитини;
- клас;
- email;
- рейтинги;
- відправку відповідей на сервер.

### 3.3 Не додавати сторонні runtime-залежності без потреби

Не додавати CDN, npm runtime packages, analytics, iframe, trackers.

Якщо залежність справді потрібна, пояснити:

- навіщо;
- чому без неї не можна;
- чи вона працює offline;
- чи вона не збирає дані;
- як вона впливає на CSP.

### 3.4 Не повертати моноліт

Не створювати один великий `script.js`.

Не переносити всю логіку в `GameScene`.

Не змішувати:

```txt
дані задач + DOM renderer + перевірка + рахунок + сцена
```

---

## 4. Технологічна позиція

Поточний стек:

```txt
HTML + CSS + vanilla JS + ES modules + DOM rendering + PWA
```

Не переводити проєкт на:

- React;
- Vue;
- Svelte;
- Phaser;
- canvas-only engine;
- TypeScript;
- bundler;
- backend;

без окремого рішення.

TypeScript або bundler можна розглядати пізніше, але не як перший крок.

---

## 5. Принцип роботи з архітектурою

Якщо треба додати нову логіку, спочатку подумати, де вона має жити:

```txt
core/       базові утиліти
app/        bootstrapping
scenes/     екрани й modal/dialog lifecycle
game/       ігрові правила, світ, NPC, яблука, колізії
tasks/      задачі, task data, task renderers, evaluators
systems/    системи рахунку або подій
ui/         HUD, тема, шрифт
i18n/       тексти
pwa/        Service Worker реєстрація
```

Не додавати код у перший файл, який трапився.

---

## 6. Правило для `GameScene`

`GameScene` уже занадто багато робить.

Не додавати в `GameScene` нові великі блоки логіки, якщо їх можна винести.

Перевага для нових модулів:

```txt
js/game/world-generator.js
js/game/spawn-rules.js
js/game/collision-system.js
js/game/apple-system.js
js/game/npc-system.js
js/game/camera-controller.js
js/game/session-state.js
```

Якщо зміна збільшує `GameScene`, агент має пояснити, чому це виправдано.

---

## 7. Правило для задач

Нові задачі додавати через `tasks/`, а не через `GameScene` або `DialogScene`.

Поточний мінімум:

```txt
task-data -> task-pools -> task-registry -> task-type
```

Бажаний напрям:

```txt
task-data -> task-validator -> task-renderer -> task-evaluator
```

Перед додаванням складної задачі прочитати `TASKS.md`.

---

## 8. Як додавати новий task type

1. Створити файл у `js/tasks/task-types/`.
2. Якщо потрібні дані — створити файл у `js/tasks/task-data/`.
3. Не використовувати `innerHTML` для змінних текстів.
4. Зареєструвати тип у `js/tasks/task-registry.js`.
5. Додати entry у `js/tasks/task-data/task-pools.js`.
6. Додати або оновити i18n-тексти.
7. Додати тест.
8. Перевірити keyboard/touch.
9. Перевірити, що `onSolved()` викликається один раз.

---

## 9. Математичні задачі з доданого прикладу

У проєкті є окремий приклад математичного скрипта з корисними ідеями:

```txt
number sequence
math pyramid
balance equation
symbol logic
magic square
math machine
```

Не копіювати цей файл як моноліт у гру.

Правильна дія:

- взяти педагогічну ідею;
- розбити на generator/renderer/evaluator;
- додати task validation;
- додати тести;
- зробити доступний DOM UI.

---

## 10. DOM і безпека

Перевага:

```js
const el = document.createElement('button');
el.textContent = label;
```

Не робити:

```js
container.innerHTML = `<button>${label}</button>`;
```

Дозволено:

```js
container.innerHTML = '';
```

для очищення контейнера.

Якщо `innerHTML` потрібен, треба мати вагому причину й escaping.

---

## 11. EventBus

Поточний `EventBus` простий. Будьте обережні з повторними subscriptions.

Бажане покращення:

- додати `off()`;
- додати `reset()`;
- або зробити systems idempotent.

Не створювати багато неописаних глобальних подій.

Нову подію документувати в `ARCHITECTURE.md`.

---

## 12. Input

`Input.init()` додає глобальні listeners.

Не викликати повторно без захисту.

Бажано зробити `Input.init()` ідемпотентним:

```js
if (this.initialized) return;
```

Не ламати:

- WASD;
- arrow keys;
- touch;
- mouse;
- keyboard cursor mode;
- input fields у задачах.

---

## 13. PWA

Якщо додається, видаляється або перейменовується файл, який потрібен offline:

1. Оновити `sw.js` `STATIC_ASSETS`.
2. Оновити `STATIC_CACHE` version.
3. Перевірити install Service Worker.
4. Перевірити offline.

Під час debugging завжди враховувати, що Service Worker може показувати старий код.

---

## 14. i18n

Усі користувацькі тексти мають бути українською.

Бажано додавати тексти в:

```txt
js/i18n/uk.js
```

Не розкидати великі тексти по модулях.

Допустимі винятки:

- короткі dev-only повідомлення;
- тимчасові console errors;
- data-driven task text, якщо task-data є джерелом педагогічного контенту.

---

## 15. CSS і UI

Перед змінами стилів прочитати `tokens.css` і `PROJECT_STANDARDS.md`.

Правила:

- використовувати design tokens;
- не додавати хаотичні кольори;

- не прибирати focus-visible;
- перевіряти mobile;

## 16. Accessibility

Не ламати:

- `aria-live` announcer;
- keyboard navigation;
- focus trap;
- modal focus;
- reduced motion;
- readable font;
- semantic buttons/inputs.

Жодна нова навчальна дія не має вимагати тільки мишу.

## 19. Переважний стиль коду

- ES modules.
- Іменовані exports, якщо модуль не має очевидного default.
- Маленькі функції.
- Чисті функції для правил і обчислень.
- DOM-операції тримати в renderer/controller, не в data.
- Без глобальних mutable singleton там, де можна передати залежності.
- Коментарі лише там, де пояснюють “чому”, а не очевидне “що”.

---

## 20. Заборонені quick fixes

Не робити:

```txt
- setTimeout як приховане вирішення race condition без пояснення
- дублювання логіки score в кількох місцях
- ручне виправлення DOM після кожного render замість нормального state flow
- hardcoded NPC-specific task у GameScene
- localStorage для прогресу
- CSS !important без крайньої потреби
- приховування помилки console.clear()
- catch(error) {} без logging у dev
```

---

## 21. Пріоритети роботи для агентів

Якщо агенту дали загальну задачу “покращити проєкт”, працювати в такому порядку:

1. Захистити session policy.
2. Зробити `Input`/`EventBus` безпечними від повторної ініціалізації.
3. Винести логіку з `GameScene`.
4. Додати task validation.
5. Додати task evaluators.
6. Додати тести для task-data.
7. Додати PWA asset check.
8. Додати нові математичні задачі.
9. Поліпшити UX фінального екрану.

Не починати з декоративних змін, якщо є архітектурні ризики.

## 23. Definition of done для агента

Задача виконана, якщо:

- [ ] зміна відповідає документації;
- [ ] немає порушення privacy/session policy;
- [ ] архітектура не стала більш монолітною;
- [ ] нові задачі не пришиті до `GameScene`;
- [ ] UI доступний;
- [ ] PWA враховано;
- [ ] тести або manual checklist виконані;
- [ ] документація оновлена за потреби;
- [ ] агент чесно повідомив про неперевірені частини.
