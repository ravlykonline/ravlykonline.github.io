# SECURITY.md — безпека, приватність і дитячий режим

Цей документ описує модель безпеки проєкту «Равлик-мандрівник».

Гра призначена для дітей і має працювати на спільних шкільних пристроях. Тому безпека тут означає не лише захист від технічних атак, а й захист приватності дитини, зрозумілість поведінки та відсутність прихованого збору даних.

---

## 1. Security posture

Поточний проєкт — статичний сайт без бекенду.

Це означає:

- немає серверної авторизації;
- немає бази даних;
- немає реєстрації;
- немає платежів;
- немає обробки персональних даних;
- немає передачі відповідей дітей на сервер;
- немає потреби в cookies;
- немає потреби в токенах доступу.

Це сильна сторона. Її треба зберегти.

---

## 2. Основна політика приватності

Гра не повинна збирати персональні дані дітей.

Заборонено без окремого рішення:

- ім'я дитини;
- прізвище;
- клас;
- школа;
- email;
- телефон;
- фото;
- голос;
- геолокація;
- IP-аналітика на рівні продукту;
- історія відповідей;
- індивідуальний рейтинг;
- профіль користувача.

Якщо в майбутньому з'явиться потреба в облікових записах, це має бути окремий проєктний етап із юридичною, технічною й педагогічною оцінкою.

---

## 3. Session policy

Гра є сесійною.

Дозволено зберігати:

- тему інтерфейсу;
- режим читабельного шрифту;
- інші неособисті UI-налаштування.

Заборонено зберігати між reload/open:

- кількість яблук;
- кількість зірочок;
- виконані завдання;
- імена дітей;
- історію відповідей;
- час проходження;
- індивідуальний результат.

Причина: спільні шкільні комп'ютери.

Наступна дитина не повинна бачити прогрес попередньої.

---

## 4. Threat model

### 4.1 Основні активи

Що треба захищати:

- приватність дітей;
- цілісність навчального досвіду;
- стабільність гри;
- відсутність небажаного стороннього коду;
- передбачуваність PWA-кешу;
- репутацію освітнього сайту.

### 4.2 Неактиви

Що не є секретом:

- правильні відповіді до задач;
- структура рівнів;
- кількість яблук;
- логіка нарахування зірочок.

Це клієнтська браузерна гра. Будь-яка відповідь, яка є в JavaScript, може бути знайдена через DevTools. Не треба будувати псевдозахист від списування.

### 4.3 Основні ризики

| Ризик | Важливість | Коментар |
|---|---:|---|
| XSS через небезпечний HTML | Висока | Особливо якщо з'являться зовнішні дані або редактор задач |
| Сторонні CDN | Висока | Підвищують supply-chain ризик |
| Аналітика без рішення | Висока | Може порушити дитячу приватність |
| PWA показує старий код | Середня | Може приховувати виправлення багів |
| Дублювання event listeners | Середня | Може давати подвійні бали або дивні стани |
| Збереження прогресу на спільному ПК | Висока | Приватність і UX-проблема |
| Некоректні task-data | Середня | Ламає навчальний досвід |
| Clickjacking | Низька/середня | Варто закрити через headers |

## 5. Content Security Policy

Для production бажано виставити такі headers на Cloudflare або іншому хостингу:

```txt
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'; form-action 'none'; manifest-src 'self'; worker-src 'self'
Referrer-Policy: no-referrer
X-Content-Type-Options: nosniff
Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=(), magnetometer=(), gyroscope=(), accelerometer=(), clipboard-read=(), clipboard-write=()
```

Якщо в майбутньому з'являться шрифти, картинки або API з інших доменів, CSP треба оновлювати явно й документувати причину.

## 6. Cloudflare Pages headers

Якщо сайт хоститься через Cloudflare Pages, можна додати файл:

```txt
_headers
```

Приклад:

```txt
/*
  Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'; form-action 'none'; manifest-src 'self'; worker-src 'self'
  Referrer-Policy: no-referrer
  X-Content-Type-Options: nosniff
  Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(), usb=(), magnetometer=(), gyroscope=(), accelerometer=(), clipboard-read=(), clipboard-write=()
```

Якщо через таку CSP щось ламається, не треба одразу послаблювати весь policy. Спочатку треба з'ясувати, який саме ресурс потребує доступу й чи він справді потрібен.

## 7. HTML insertion policy

Стандарт безпеки для UI:

- створювати елементи через `document.createElement`;
- вставляти текст через `textContent`;
- використовувати `innerHTML = ''` лише для очищення контейнерів;
- не вставляти задачі, підказки, імена NPC або будь-які змінні тексти через HTML-шаблони;
- якщо використовується `innerHTML`, обов'язково пояснити причину й гарантувати escaping.

Добре:

```js
const button = document.createElement('button');
button.textContent = choice.label;
```

Погано:

```js
container.innerHTML = `<button>${choice.label}</button>`;
```

Навіть якщо зараз усі дані локальні й контрольовані, правило треба тримати, бо в майбутньому задачі можуть редагуватися окремо.

## 8. Task data security

Задачі не є секретом. Але вони мають бути валідними.

Потрібні перевірки:

- `task.type` існує;
- відповідний renderer/evaluator існує;
- правильна відповідь є серед варіантів;
- немає дублікатів id;
- reward валідний;
- prompt не порожній;
- instructions не порожні;
- текст не містить HTML, якщо renderer не очікує HTML;
- numeric range відповідає віку.

Рекомендовано додати:

```txt
js/tasks/task-contracts.js
js/tasks/task-validator.js
tests/task-data-validation.js
```

## 9. Service Worker security

`sw.js` кешує статичні файли. Це зручно для PWA, але має ризики:

- браузер може показувати стару версію гри;
- виправлення безпеки може не з'явитися одразу;
- список `STATIC_ASSETS` може застаріти;
- відсутній файл у cache list може ламати install.

Правила:

1. Після зміни файлів із `STATIC_ASSETS` оновлювати cache version.
2. Перед release перевіряти, що всі файли зі списку існують.
3. У dev-режимі вміти швидко вимкнути Service Worker.
4. Не кешувати персональні дані, бо їх не повинно бути.
5. Не кешувати сторонні ресурси без рішення.

Детальніше: [PWA.md](PWA.md).

## 10. Third-party dependencies

Поточна бажана політика:

```txt
no third-party runtime dependencies by default
```

Не додавати без окремого рішення:

- CDN scripts;
- Google Analytics;
- Facebook Pixel;
- Hotjar/Clarity/session replay;
- сторонні iframe;
- рекламні scripts;
- remote fonts;
- npm-пакети для runtime без потреби.

Якщо залежність усе ж потрібна, треба додати в PR/коміт:

- для чого вона;
- чи можна без неї;
- які альтернативи;
- чи працює вона offline;
- чи збирає дані;
- як вона впливає на CSP;
- як вона впливає на дитячу приватність.

---

## 11. Analytics policy

За замовчуванням аналітика заборонена.

Якщо в майбутньому потрібна аналітика, вона має бути:

- privacy-first;
- без cookies;
- без user id;
- без індивідуальних профілів;
- без запису відповідей дітей;
- без session replay;
- з окремою політикою приватності.

Для поточної гри аналітика не потрібна.

---

## 12. LocalStorage / SessionStorage policy

### Дозволено в `localStorage`

Тільки неособисті UI-налаштування:

- тема;
- читабельний шрифт;
- звук;
- інші налаштування доступності.

### Дозволено в `sessionStorage`

Можна розглядати тільки для поточної вкладки й лише якщо це не суперечить сесійній політиці. Наприклад, тимчасовий debug seed або UI state.

### Заборонено

- прогрес рівнів;
- відповіді;
- ім'я дитини;
- результати;
- історію проходження.

---

## 13. Error handling

У production дитина не має бачити технічні помилки.

Погано:

```txt
Unknown task type: compare_magic_42
```

Добре:

```txt
Ой, це завдання загубилося. Спробуй інше.
```

У dev-консолі технічну помилку залишати можна й потрібно.

Рекомендація:

```js
try {
    TaskRegistry.renderTask(...);
} catch (error) {
    console.error(error);
    renderSafeTaskFallback();
}
```

---

## 14. Input safety

Поточна гра майже не має відкритого введення тексту. Якщо з'являться поля вводу:

- не приймати HTML;
- обмежувати довжину;
- використовувати `textContent` для повторного показу;
- не зберігати введення між сесіями;
- не відправляти введення на сервер.

Для математичних задач numeric input має:

- приймати тільки потрібні символи;
- мати `inputmode="numeric"`;
- мати зрозумілий `aria-label`;
- мати обмеження довжини;
- не створювати raw HTML.

---

## 15. Clickjacking

Production headers мають містити:

```txt
frame-ancestors 'none'
```

Це забороняє вбудовувати гру в чужі iframe.

Якщо в майбутньому гру треба вбудовувати в офіційний сайт, дозволений домен треба додати явно:

```txt
frame-ancestors 'self' https://example.edu.ua
```

---

## 16. Permissions Policy

Гра не потребує:

- камери;
- мікрофона;
- геолокації;
- USB;
- платежів;
- сенсорів;
- clipboard.

Тому ці можливості мають бути заборонені через `Permissions-Policy`.

---

## 17. Supply-chain policy

Проєкт має залишатися максимально незалежним.

Перед додаванням npm-пакета або CDN треба відповісти:

1. Чи можна зробити це 20–50 рядками власного коду?
2. Чи пакет підтримується?
3. Чи він потрібен у runtime, чи лише для dev?
4. Чи він не збирає дані?
5. Чи не ламає offline?
6. Чи не послаблює CSP?
7. Чи не ускладнює аудит?

---

## 18. Security checklist перед release

Перед публікацією перевірити:

- [ ] немає сторонніх runtime scripts;
- [ ] немає аналітики;
- [ ] немає форм персональних даних;
- [ ] прогрес не зберігається між reload;
- [ ] `localStorage` використовується лише для UI preferences;
- [ ] Service Worker cache version оновлена;
- [ ] усі `STATIC_ASSETS` існують;
- [ ] CSP не ламає гру;
- [ ] headers налаштовані;
- [ ] задачі проходять validation;
- [ ] модальні вікна не ламають focus trap;
- [ ] keyboard navigation працює;
- [ ] reduced motion працює;
- [ ] DevTools console не містить критичних помилок.

---

## 19. Що робити при security bug

Якщо знайдено проблему:

1. Не приховувати її.
2. Визначити, чи є ризик для дітей або приватності.
3. Якщо є ризик приватності — тимчасово вимкнути проблемну функцію.
4. Додати regression test.
5. Оновити `SECURITY.md`, якщо правило не було описане.
6. Оновити Service Worker cache version.
7. Перевірити production після деплою.

---

## 20. Головна безпекова позиція

Найкращий захист цього проєкту — простота:

- немає бекенду;
- немає акаунтів;
- немає персональних даних;
- немає стороннього runtime коду;
- немає складної авторизації;
- немає прихованого збереження дитячої активності.

Цю простоту потрібно зберігати як архітектурну перевагу.
