# Поточний стан проєкту «Равлик-бродилка»

> Оновлено: 2026-04-18

## Продуктове бачення

Проєкт розвивається як сесійна дослідницька гра-бродилка для дітей 5-7 років. Дитина керує Равликом, досліджує територію, збирає яблука й зустрічає звірів, які дають короткі візуальні завдання.

Головна ігрова петля:

```txt
дитина заходить у гру
  -> керує Равликом
  -> досліджує поле
  -> збирає яблука
  -> підходить до звірів
  -> пробує короткі візуальні завдання
  -> отримує зірочки за успіх
  -> може вийти із завдання і йти далі
```

Кожне нове відкриття або перезавантаження браузера має створювати нову сесію: яблука розміщуються інакше, доступні звірі та їхні завдання можуть змінюватися, прогрес попередньої дитини не відновлюється.

Цільова аудиторія ще не читає або читає невпевнено, тому гра має покладатися на поле, персонажів, предмети, форми, кольори, прості дії та короткі підказки. Тексти мають бути мінімальними. Задачі мають бути переважно з об'єктами, формами, парами, послідовностями, групами й простими візуальними закономірностями, а не з довгими умовами чи складними математичними розв'язками.

Поле гри є головним екраном. HUD має залишатися простим: кількість яблук, кількість зірок і дуже короткий контекст поруч зі звіром.

## Призначення

Проєкт розвивається як урокова сесійна гра для дітей 1–2 класу. На одному комп'ютері можуть грати різні діти, тому після перезавантаження сторінки або повторного відкриття браузера має стартувати нова гра без відновлення попереднього проходження.

### Архітектура

- Код розбитий на модулі замість одного великого `script.js`.
- Є окремі шари для сцен, систем, базових утиліт, i18n, UI-контролерів і PWA.

- Bootstrap застосунку: [js/app/bootstrap.js](C:/Users/artem/Documents/GitHub/ravlykonline.github.io/game/js/app/bootstrap.js)

### Гра

- Равлик плавно рухається з м'якшим прискоренням, сповільненням і розворотом.
- Яблука збираються у світі на DOM-елементах.
- NPC мають випадкове завдання на кожну нову гру.
- Для NPC уже працює базова система головоломок із кількома варіантами в кожному типі.
- Уже є типи задач: продовження послідовності, пошук зайвого предмета, впорядкування чисел за зростанням і спаданням, порівняння множин, лічба предметів, просте додавання, візерунки з фігур, просте віднімання й логічні пари.
- Після виконання завдання NPC дитина отримує зірочки.
- Перший екран містить onboarding перед стартом ігрової сцени.
- Взаємодія з NPC відбувається усвідомлено: потрібно підійти ближче і натиснути `Enter` або клікнути.

### HUD та UI

- HUD винесений у керований шар через [js/ui/hud-controller.js](C:/Users/artem/Documents/GitHub/ravlykonline.github.io/game/js/ui/hud-controller.js).
- У HUD уже є:
- статистика яблук і зірочок;
- окремий сесійний статус;
- окрема основна ціль;
- окремий контекстний блок про те, що відбувається поруч;
- перемикач читабельного шрифту;
- перемикач світлої/темної теми.

### Доступність

- Є keyboard navigation, `skip-link`, `aria-live`, фокус-трап у діалозі.
- Світ побудований на DOM, а не на canvas.
- Підтримано `prefers-reduced-motion`.
- Ключові тексти централізовано в i18n.

### PWA

- Є `manifest.json`, `sw.js`, `offline.html`, іконки.
- Service Worker кешує основні файли застосунку.
- Сесійна політика збережена: офлайн-режим доступний, але нове відкриття гри стартує з чистого стану.

### Тести

- Є браузерні unit/integration тести.
- Є окремий тест на артефакти кодування українського тексту.
- Покриті критичні правила руху, випадковий вибір завдань, `taskRegistry`, theme mode і HUD controller.
- Інтеграційний контур уже перевіряє основний цикл `NPC -> puzzle -> stars -> HUD`.

## Поточні принципи, які вже зафіксовані

- Гра є сесійною, а не прогресивною між запусками.
- Після reload або нового відкриття браузера стартує нова гра.
- Це не вважається втратою прогресу і не повинно "виправлятися" автозбереженням без окремої teacher-facing вимоги.
- Розробка ведеться саме в межах папки `game`: це окрема гра-бродилка, не AST/parser/runtime середовище з кореня репозиторію.
- Основна цільова аудиторія — діти 5-7 років, тому нові UX-рішення мають зменшувати текст і підтримувати гру через візуальні об'єкти.
- Дитина має мати право не виконати завдання: діалог зі звіром можна закрити без покарання й продовжити дослідження.
- Довгострокова ціль світу — приблизно 20-25 звірів, рівномірно розкиданих по полі й доступних для взаємодії.
- Рішення в UI і логіці мають мінімізувати технічний борг через окремі контролери, токени та модулі.
- Усе нове бажано додавати через окремі модулі, а не через розростання `GameScene`.

## План розвитку

### 0. Продуктовий напрям

Найближчий розвиток має підтримати гру як легку бродилку:

- спростити перший вхід у гру й onboarding;
- тримати HUD мінімальним: яблука, зірки, короткий контекст;
- розвинути session generation, щоб кожен reload давав нове поле, яблука й завдання;
- підготувати spawn rules для 20-25 звірів без накладання й недоступних позицій;
- розвивати задачі як короткі візуальні взаємодії для дітей 5-7 років;
- не перетворювати гру на тестову платформу з довгими текстами.

### 1. Що вже закладено як фундамент

- Є `taskPoolId` у NPC замість жорстко пришитих текстових реплік.
- Є `taskRegistry`, який випадково підтягує структуровану задачу з пулу.
- Є окремий шар `task-data`, де лежать варіанти задач і конфігурація пулів.
- Є окремі `task-types`, які можна розширювати без переписування `GameScene`.
- Є drag-free puzzle dialog, який рендерить різні типи взаємодії через кнопки й selection model.

### 2. Наступний крок для системи задач

База вже працює, тому наступний етап — не "заново зробити головоломки", а розширити їх без руйнування поточної моделі.

Рекомендований напрям:

- додати `task-evaluators`, якщо логіка типів задач почне ускладнюватися;
- далі розділити варіанти задач не лише по типах, а й по темах, складності та навчальних блоках;
- додати рівні складності й тематичні набори задач;
- дати кожному NPC власну освітню спеціалізацію.

### 3. Розширення типів завдань і головоломок

Поточна база вже є, тому далі потрібно розширювати спектр навчальних сценаріїв.

Рекомендована цільова модель:

- `taskDefinition` описує тип задачі, дані й правила перевірки.
- `taskRenderer` відповідає за візуальне представлення конкретного типу задачі.
- `taskEvaluator` перевіряє відповідь і повертає результат.
- `taskPool` зберігає набори задач по темах, складності й NPC.

Мінімальний формат задачі, який варто закласти:

```js
{
  id: 'compare-sets-01',
  type: 'compare-sets',
  difficulty: 'easy',
  promptKey: 'tasks.compareSets.prompt',
  reward: { stars: 1 },
  payload: {
    leftItems: 4,
    rightItems: 6,
    question: 'more'
  }
}
```

Бажані наступні типи задач для віку 5-7 років:

- `visual-groups`
- `missing-item`
- `choose-rule`
- `memory-pairs`
- `shadow-match`
- `shape-sequence`
- `same-different`
- `visual-magic-square`

### 4. Розширити NPC-модель

NPC мають перейти від одного `activeTask` до моделі на основі ролі та пулу задач.

Бажаний напрям:

- у кожного NPC є `taskPoolId`;
- NPC може тягнути випадкове завдання відповідного типу;
- у майбутньому NPC може мати тему: лічба, логіка, послідовності, форми.

Приклад:

```js
{
  id: 'mouse_1',
  type: 'mouse',
  nameKey: 'npc.mouseName',
  taskPoolId: 'logic.beginner'
}
```

### 5. Розвинути puzzle dialog

Базовий puzzle dialog уже працює як універсальний контейнер, але надалі йому потрібні ширші сценарії для різних типів контенту:

- текст;
- вибір варіанту;
- візуальна множина;
- послідовність фігур;
- простий drag-free interaction через кнопки або selection model.

Тут важливо не йти в drag-and-drop за замовчуванням, щоб не втратити доступність.

### 6. Посилити тестову базу

Наступні тести варто додати або посилити першими:

- тести на генерацію випадкових задач без некоректних станів;
- тести на перевірку правильної/неправильної відповіді;
- інтеграційні сценарії на кілька різних типів головоломок, а не лише один happy path;
- smoke-тести на рендер конкретних типів головоломок.

### 7. Окремо подумати про дитячий UX

Наступні кроки по UX:

- м'якіший прогресний наратив у HUD;
- візуальна рамка для активного NPC або задачі;
- зрозумілий екран успіху після головоломки;
- опціональний тихий звук для збору яблука та зірочки;
- ще коротший і наочніший onboarding для першого руху.

## Що не робити без потреби

- Не повертатися до монолітного скрипта.
- Не зберігати прогрес між сесіями на shared computer.
- Не змішувати рендер задач, перевірку відповідей і дані задач в одному файлі.
- Не прив'язувати нові типи задач жорстко до `GameScene`.

## Найближчий рекомендований пакет робіт

1. Зафіксувати технічну стабільність сесії: `Input`, `EventBus`, `ScoreSystem` не мають дублювати listeners при повторному запуску.
2. Додати `SessionState`, який описує поточну сесію без збереження між reload.
3. Винести `world-generator` і `spawn-rules`, щоб підготувати випадкове розміщення яблук і 20-25 звірів.
4. Спростити onboarding/HUD так, щоб поле гри стало головним.
5. Додати `task-validator` і поступово розвивати візуальні задачі для дітей 5-7 років.
6. Розширити інтеграційні тести: reload reset, випадкові задачі, кілька NPC, відсутність повторної зірки.
