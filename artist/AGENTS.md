# AGENTS.md — інструкції для Codex, Claude Code та інших AI-агентів

Цей документ є головним технічним завданням для агентів, які будуть змінювати код Ravlyk Artist. Перед будь-якими змінами прочитати цей файл, `README.md`, `ARCHITECTURE.md`, `LEVELS.md`, `SECURITY.md`, `ACCESSIBILITY.md`, `TESTING.md` і `TASKS.md`.

---

## 1. Контекст проєкту

Ravlyk Artist — дитячий браузерний редактор алгоритмічного малювання у стилі Code.org Artist. Діти складають програму з блоків, запускають її, а Равлик малює шлях.

Проєкт має бути:

- простим;
- україномовним;
- без реєстрації;
- без backend;
- без персональних даних;
- offline-first;
- доступним без drag-and-drop;
- придатним для використання у школі.

---

## 2. Найважливіше правило

Production-код — це модульна версія:

```text
index.html
css/*.css
js/main.js
js/core/*.js
js/data/*.js
js/ui/*.js
```

Файли `script.js` і `style.css` у корені — legacy. Не додавати туди нову логіку. Першим завданням треба перенести їх у `legacy/` або видалити після перевірки.

---

## 3. Заборонено

Не робити:

- не додавати нові фічі в `script.js` або `style.css`;
- не додавати backend;
- не додавати реєстрацію;
- не збирати персональні дані дітей;
- не додавати сторонні JS-CDN без окремого обґрунтування;
- не робити drag-and-drop єдиним способом взаємодії;
- не використовувати `innerHTML` для текстів, які можуть бути даними;
- не запускати програму без execution limit;
- не зберігати прогрес у `localStorage` за замовчуванням;
- не змішувати grid-logic і turtle-logic без явного `mode`;
- не змінювати дизайн хаотично поза CSS-токенами;
- не залишати англомовні UI-рядки.

---

## 5. Перший обов’язковий пакет змін

Якщо агент починає з чистого стану, виконати в такому порядку:

1. Створити `legacy/` і перенести туди `script.js`, `style.css`.
2. Переконатися, що `index.html` не посилається на legacy-файли.
3. Додати `package.json` з `type: module` і командою `npm test`.
4. Перекласти UI в `index.html`, `js/data/blocks.js`, `js/data/lessons.js`, `js/core/goals.js`, `js/ui/render.js`.
5. Винести UI-рядки в `js/data/i18n.uk.js` або хоча б не додавати нові англомовні рядки.
6. Замінити `instructionHtml` на structured instruction або безпечний renderer.
7. Додати execution limits для repeat.
8. Додати кнопку `Зупинити` під час виконання.
9. Додати `exact-path` у goal evaluator.
10. Оновити тести.

## 6. Архітектурний напрям

Поточний код не треба переписувати повністю за один раз. Але кожна зміна має рухати проєкт у бік такої архітектури:

```text
data -> domain -> runtime -> state -> ui
```

Дані рівнів не мають залежати від DOM.

Domain-логіка має бути тестованою без браузера.

Runtime має генерувати події, а UI має їх рендерити.

State має змінюватися через action/helper-функції, а не випадковими мутаціями в різних частинах UI.

---

## 7. Правила для JavaScript

- Використовувати ES-модулі.
- Не використовувати глобальні змінні, крім контрольованого state-модуля.
- Нову чисту логіку покривати unit-тестами.
- DOM-код тримати в `ui/`.
- Runtime/engine не має напряму працювати з DOM.
- Не ловити помилки порожнім `catch`.
- Повідомлення для дітей — дружні, без stack trace.
- Усі нові функції мають мати зрозумілі назви.
- Складні функції розбивати.

---

## 8. Правила для CSS

- Використовувати токени з `css/tokens.css`.
- Не додавати випадкові hardcoded кольори без потреби.
- Не ламати responsive layout.
- Кнопки мають бути достатні для touch.
- Focus state має бути видимим.
- Reduced motion має підтримуватися.
- Не ховати важливі кнопки за горизонтальним scroll.

---

## 9. Правила для рівнів

- Усі тексти українською.
- Рівень має бути декларативним.
- Toolbox має містити тільки існуючі block types.
- Обмеження мають бути в `constraints`, а не тільки в тексті.
- Success checker має бути чесним.
- Якщо завдання просить “не більше 2 блоків”, це має перевірятися кодом.
- Кожен новий рівень має мати хоча б одну підказку.
- Не додавати сирий HTML в інструкції.

---

## 10. Правила для accessibility

Кожна нова взаємодія має мати keyboard alternative.

Якщо додається drag-and-drop фіча, паралельно додати:

- кнопку;
- keyboard shortcut;
- aria-label;
- live announcement.

Модальні вікна мають керувати фокусом.

Canvas має мати текстовий опис поточного стану.

---

## 11. Правила для безпеки

- Не додавати user-controlled HTML.
- Не додавати cookies.
- Не додавати analytics без окремого рішення.
- Не додавати сторонні скрипти.
- Не зберігати персональні дані.
- Якщо додається storage, за замовчуванням використовувати `sessionStorage`.
- Якщо додається `localStorage`, це має бути opt-in.

---

## 12. Обов’язкові тести після змін

Після кожної зміни запускати:

```bash
node tests/art.core.test.js
```

Після додавання `package.json`:

```bash
npm test
```

Якщо змінювали рівні:

```bash
npm run validate:levels
```

Якщо змінювали UI:

- вручну пройти перший рівень;
- перевірити reset;
- перевірити keyboard-only scenario;
- перевірити mobile viewport.
