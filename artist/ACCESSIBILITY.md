# ACCESSIBILITY.md — доступність і дитячий UX Ravlyk Artist

Цей документ описує, що треба реалізувати, щоб Ravlyk Artist був зручним для дітей, працював на планшетах, не залежав лише від drag-and-drop і мав базову відповідність WCAG.

---

## 1. Головний принцип

Усі ключові дії мають працювати щонайменше трьома способами:

1. мишкою;
2. дотиком на планшеті;
3. клавіатурою.

Drag-and-drop може бути додатковою зручною взаємодією, але не єдиною.

---

## 2. Поточні проблеми

1. Перестановка блоків залежить від drag-and-drop.
2. Немає кнопок `Вгору`, `Вниз`, `Видалити`, `Додати всередину repeat`.
3. Canvas/SVG майже повністю приховані від screen reader через `aria-hidden="true"`.
4. `canvas-status` повідомляє лише загальний стан, але не описує шлях і ціль.
5. UI частково англійський.
6. Немає повної keyboard navigation для workspace.
7. Drop zones мають `aria-hidden="true"`, але немає альтернативи для користувачів клавіатури.
8. Модальні вікна треба перевірити на focus trap і повернення фокусу.

---

## 3. Мінімальні вимоги MVP

### 3.1. Palette

Кожен блок у палітрі має бути `<button>`.

Дія:

- click/tap додає блок у кінець workspace;
- keyboard `Enter` або `Space` також додає блок;
- після додавання фокус переходить на доданий блок або залишається в палітрі залежно від обраної UX-моделі.

### 3.2. Workspace

Кожен блок у workspace має мати доступні дії:

- `Вгору`;
- `Вниз`;
- `Всередину` або `Додати в repeat`;
- `Назовні`, якщо блок у repeat;
- `Видалити`;
- для repeat — зміна кількості.

Кнопки можуть бути компактними візуально, але доступними для screen reader.

### 3.3. Keyboard shortcuts

MVP:

| Комбінація | Дія |
|---|---|
| `Tab` | перехід між зонами |
| `Enter` / `Space` | активувати кнопку/блок |
| `Delete` / `Backspace` | видалити обраний блок |
| `ArrowUp` | перемістити блок вище |
| `ArrowDown` | перемістити блок нижче |
| `Ctrl+Z` | скасувати |
| `Ctrl+Y` | повернути |
| `Ctrl+Enter` | запустити програму |
| `Escape` | закрити модальне вікно / скасувати вибір |

## 4. Canvas accessibility

Потрібен текстовий опис поля.

Додати DOM-регіон:

```html
<section aria-label="Поле малювання">
  <p id="canvas-goal" class="sr-only"></p>
  <p id="canvas-position" class="sr-only"></p>
  <ol id="canvas-trail-list" class="sr-only"></ol>
</section>
```

Після кожного кроку оновлювати:

```text
Равлик у колонці 4, рядку 5. Намальовано 3 відрізки з 4.
```

Для успіху:

```text
Завдання виконано. Равлик намалював потрібний шлях.
```

Для помилки:

```text
Потрібний шлях ще не завершено. Перевір кількість кроків і напрямок.
```

---

## 5. Live regions

Використовувати два live-регіони:

```html
<div id="feedback" aria-live="polite"></div>
<div id="sr-announcer" class="sr-only" aria-live="polite" aria-atomic="true"></div>
```

Не оголошувати кожну дрібну анімацію, якщо це створює шум. Для screen reader достатньо важливих подій:

- програма запущена;
- крок виконано;
- Равлик уперся в край;
- програма завершена;
- завдання виконано;
- завдання ще не виконано;
- блок додано/видалено/переміщено.

---

## 6. Модальні вікна

Для success/reset dialogs потрібно:

- `role="dialog"`;
- `aria-modal="true"`;
- `aria-labelledby`;
- `aria-describedby`;
- focus trap;
- фокус на першій логічній кнопці;
- `Escape` закриває, якщо це не critical action;
- після закриття фокус повертається на кнопку, що відкрила діалог.

Для reset confirmation:

- фокус за замовчуванням краще ставити на `Скасувати`, а не на destructive action.

---

## 7. Рух і анімації

Підтримати `prefers-reduced-motion`.

Якщо користувач обрав reduced motion:

- не робити довгі плавні переходи;
- не використовувати bouncing/pulsing;
- виконувати програму швидше або покроково без анімації;
- не показувати інтенсивні ефекти успіху.

---

## 8. Візуальна доступність

Мінімальні вимоги:

- достатній контраст тексту;
- focus outline видимий;
- кнопки не менше 40×40 px для touch;
- не покладатися тільки на колір;
- активний рівень має відрізнятися не тільки кольором, а й текстом/іконкою;
- помилка має мати текст, не лише червоний колір.

---

## 9. Українська мова

Усі UI-рядки мають бути українською:

```text
Lesson -> Рівень
Blocks -> Команди
Workspace -> Програма
Run -> Запустити
Reset -> Очистити
Code -> Код
When run -> Коли запустити
Task -> Завдання
Next lesson -> Наступний рівень
```

Дивись `ARCHITECTURE.md` про `i18n.uk.js`.

## 10. Дитячий UX

### Повідомлення мають бути дружні

Погано:

```text
Invalid path.
```

Добре:

```text
Майже вийшло. Перевір, чи Равлик пройшов усі потрібні клітинки.
```

### Не перевантажувати екран

- Не показувати одразу всі складні налаштування.
- Базові дії тримати видимими.
- Підказки показувати за запитом.
- Код показувати через окрему кнопку.

### Дати дитині контроль

Обов’язкові кнопки:

- `Запустити`;
- `Зупинити`, коли програма виконується;
- `Очистити`;
- `Скасувати`;
- `Повернути`;
- `Підказка`.

## 11. Чекліст доступності

- [ ] Усі блоки палітри — кнопки.
- [ ] Блоки можна додавати без drag-and-drop.
- [ ] Блоки можна переміщати без drag-and-drop.
- [ ] Блоки можна вкладати в repeat без drag-and-drop.
- [ ] Є keyboard shortcuts.
- [ ] Є видимий focus outline.
- [ ] Є текстовий опис canvas.
- [ ] Live regions не створюють зайвого шуму.
- [ ] Модалки мають focus trap.
- [ ] Escape працює очікувано.
- [ ] Reset confirmation фокусується на безпечній дії.
- [ ] Підтримано reduced motion.
- [ ] UI повністю українською.
- [ ] Кнопки достатнього розміру для планшета.
